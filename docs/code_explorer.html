main.py Random Moving Squares (Pygame Game Loop)
Function Call Relationships
Shows how functions invoke each other. The main() function orchestrates the game loop, calling update_square and draw_square for each square, while handle_input processes events and returns updated state.
External Dependencies
Project imports pygame (game framework) and stdlib modules: random (RNG), dataclasses (Square type), math (distance calculations). All external—no internal modules.
Main Game Loop Execution (Frame-by-Frame)
Each frame: (1) poll events and handle input, (2) clear screen, (3) update each square's position and apply behaviors, (4) draw all squares, (5) display overlay, (6) flip display and tick clock. Paused flag skips step 3.
Data Flow: squares List (Primary Game State)
The squares list is created in main(), updated via handle_input() (reset on R key), modified per-frame in update_square(), drawn via draw_square(), and rendered by pygame. It is the core mutable state shared across all phases.
Data Flow: paused Control Flag
Paused flag toggled on SPACE key via handle_input(), controls whether update_square is called. Provides a simple pause mechanism without stopping the render loop or clock.
✅ Good Patterns (3)
1. Dataclass for Structured Data
Organizes square position, velocity, color, and size in a single immutable-friendly container (Square @dataclass), eliminating manual __init__ boilerplate.
Improves readability and maintainability. Type checkers can verify field accesses. Reduces bugs from swapped parameters.
Scales well: adding a new attribute (e.g., rotation angle) is a one-line change. Encourages consistent data representation across functions.
2. Single-Responsibility Functions
Each function does one job: create_random_square() builds a square, apply_flee_behavior() handles predator-prey logic, update_square() moves and bounces, draw_square() renders. No function mixes concerns.
Easier to test, debug, and reuse. Changing predator logic requires editing only apply_flee_behavior().
Enables composition: update_square() orchestrates random_nudge() and apply_flee_behavior() in sequence. Supports future refactors (e.g., extracting behaviors into a separate module).
3. Module-Level Constants for Tuning
Game parameters (WINDOW_WIDTH, FPS, colors, sizes) defined at the top as constants. No magic numbers scattered throughout the code.
Game designers can tweak FPS or SQUARE_COUNT without digging into function logic. Reduces copy-paste mistakes.
Signals intent: these are the "knobs" players/developers interact with. Makes the code self-documenting.
⚠️ Potential Issues (2)
1. Hardcoded Speed Mapping Numbers
Line 63: max_speed = 40 - (size - MIN_SQUARE_SIZE) * (40 - 5) / ... uses magic numbers 40 and 5 without names. Unclear what they represent.
Reviewers and future maintainers can't easily understand the speed formula. Changing the formula requires finding all uses of these numbers.
Solution: Extract to named constants MAX_SPAWN_SPEED = 40, MIN_SPAWN_SPEED = 5 at module level, then use them in the formula.
2. Dead Commented Code (UP/DOWN Keys)
Lines 172–175 comment out UP/DOWN keyboard handlers. Confuses readers: is this a TODO? Was it removed? Is it intentional?
Increases cognitive load. Future developers might reimplement the same feature, not realizing the code exists hidden.
Best practice: remove commented code from production. Use version control (git) to recover it if needed. If it's an intentional TODO, mark it clearly with a TODO comment.

✅ Strengths (2)
Concrete Generic Types
Uses tuple[int, int, int] for RGB colors and list[Square] for collections—not bare tuple or list. Type checkers can verify RGB tuple operations.
Catches mistakes: passing a 4-tuple (RGBA) where RGB is expected, or mixing Square and str in a list. IDE autocomplete knows Square's fields.
Future developers immediately understand data shapes without reading docstrings. Reduces integration bugs.
Consistent Annotation Coverage
Every function from create_random_square() to handle_input() has full annotations on parameters and returns. Even complex returns like tuple[bool, bool, int, list[Square]] are explicit.
Static type checkers (mypy, pyright) can validate the entire codebase. Enables IDE refactoring tools to rename/move functions safely.
Sets a professional standard for the project. New contributors see the pattern and follow it naturally.
ℹ️ Observations (2)
pygame Type Coverage
Annotates pygame.Surface and pygame.event.Event directly—good. However, pygame's type stubs may not be comprehensive (depends on your IDE/mypy setup).
Consider adding local type aliases for frequently used types: ColorRGB = tuple[int, int, int] at the module level for clarity.
In larger projects, create a types.py module to centralize type definitions, keeping main.py focused on logic.
Immutability Opportunities
Square is a @dataclass but not frozen. apply_flee_behavior() and random_nudge() mutate velocity fields. Works, but mutation makes reasoning about state harder.
For learning: frozen dataclasses (frozen=True) force functions to return new objects instead of mutating. Prevents accidental side effects.
In games, performance matters; immutability has a cost. Current mutable design is pragmatic. Consider it for future teaching of functional paradigms.
